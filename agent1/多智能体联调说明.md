## 2. 三个 Agent 角色

- `historian`：历史学家视角，负责历史来源、文化背景、传播脉络、文化意义
- `craftsman`：匠人视角，负责工艺步骤、材料、技法、关键工序、传承经验
- `tourist`：游客视角，负责通俗解释、看点、体验感、理解门槛、大众价值

## 3. 已落地的正式后端接口

### `POST /api/chat`

用途：
- 发起一次多智能体问答
- 后端会调用 RAG 检索
- 然后按三个角色分别生成回答
- 最后返回聚合结果

请求体示例：

```json
{
  "sessionId": "sess_001",
  "query": "惠安女服饰有什么历史意义？",
  "heritageId": "huian_costume",
  "topK": 3,
  "agentRoles": ["historian", "craftsman", "tourist"]
}
```

### `GET /api/chat/{sessionId}`

用途：
- 查询某个会话最近一次结果
- 当前为内存缓存版，适合前后端联调

## 4. 当前返回结构

`POST /api/chat` 返回统一 `Result<T>`，`data` 为：

- `sessionId`
- `query`
- `heritageId`
- `finalAnswer`
- `agentReplies`
- `retrievals`

其中 `agentReplies` 每项包含：

- `role`
- `title`
- `content`
- `references`

其中 `retrievals` 每项包含：

- `id`
- `title`
- `score`
- `content`
- `metadata`

## 6. 当前实现特点

- 已复用现有 `RagService`
- 已复用现有 Spring AI `ChatModel`
- 当前为同步调用版本
- 当前会话结果为内存缓存，不是数据库持久化
- 当前未接入异步任务接口
- 当前未接入 WebSocket / SSE 多 Agent 流式输出

## 7. 前端联调建议

前端现阶段只需要先对接：

1. `POST /api/chat`
2. `GET /api/chat/{sessionId}`

前端显示建议：

- 上方展示 `finalAnswer`
- 中间分栏展示三个 Agent 的 `title + content`
- 下方折叠展示 `retrievals`

## 8. 后续建议

建议下一步继续补这三块：

1. 异步任务接口：`POST /api/task`、`GET /api/task/{taskId}`
2. 会话持久化：把当前内存缓存改为 Redis 或数据库
3. 流式输出：增加 WebSocket 或 SSE，支持前端实时展示生成过程
