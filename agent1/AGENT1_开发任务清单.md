# Agent1 开发任务清单

## 角色定位
Agent1 是本项目的多智能体主负责人，核心目标不是准备答辩，而是完成多智能体能力的后端实现，并打通与知识库、模型调用、异步任务、前端接口之间的整体链路。

你负责的不是单点功能，而是整条主链路：
- 用户发起问题
- 系统分发给多个 Agent
- Agent 基于 RAG 检索知识
- 后端协调生成回复
- 前端获得可展示结果

## 当前总目标
完成一个最小可运行的多智能体系统 Demo，至少满足以下要求：
- 支持 1 个用户会话
- 支持多个 Agent 参与回答
- 支持知识库检索增强
- 支持基本的任务状态管理
- 支持前端调用和结果展示

## 你的核心职责

### 1. 搭建多智能体后端框架
基于文档中的技术路线，优先使用 `SpringBoot` 组织后端工程。

你需要完成：
- 建立基础项目结构
- 划分 `controller`、`service`、`agent`、`rag`、`config`、`model` 等模块
- 预留配置文件用于模型、向量库、Redis 等组件接入

建议目录：

```text
src/main/java/.../
  controller/
  service/
  agent/
  rag/
  model/
  config/
  util/
```

### 2. 定义多智能体角色体系
根据计划文档，项目中的多智能体不是普通聊天机器人，而是围绕非遗文化场景设计的角色协作系统。

你至少需要定义以下 Agent：
- 历史学家 Agent：负责文化背景、历史来源、意义解读
- 匠人 Agent：负责工艺流程、技法、材料、传承细节
- 游客 Agent：负责体验视角、互动感受、通俗表达

你需要完成：
- 每个 Agent 的角色定义
- 每个 Agent 的系统提示词
- 每个 Agent 的输入输出格式
- 各 Agent 是否串行协作、并行生成或聚合回答

建议先实现统一接口：

```java
public interface AgentExecutor {
    String role();
    AgentReply reply(AgentContext context);
}
```

### 3. 设计 Agent 调度逻辑
这是你最核心的工作之一。你要决定用户问题进入系统后，多个 Agent 如何参与。

建议先做一个最小版本：
- 用户输入问题
- 调度器将同一问题分发给多个 Agent
- 每个 Agent 独立检索知识并生成回答
- 系统将多个回答聚合为统一输出

后续再升级：
- 支持 Agent 之间相互引用
- 支持二轮讨论
- 支持总结 Agent 输出最终结果

建议模块：

```java
AgentOrchestrator
AgentRegistry
AgentContextBuilder
AnswerAggregator
```

### 4. 接入大模型调用层
文档中提到可以使用 `Spring AI`。你需要把模型调用封装起来，不能让业务逻辑直接散落在各处拼 prompt。

你需要完成：
- 模型客户端配置
- Chat 调用封装
- 超时、异常、重试处理
- 统一返回结构

建议封装：

```java
public interface LlmClient {
    String chat(String systemPrompt, String userPrompt, List<String> references);
}
```

### 5. 实现 RAG 知识库能力
文档明确要求多智能体回答要基于资料，避免“胡编”。因此你必须实现检索增强生成。

最低要求：
- 收集非遗相关文本资料
- 对文档分段切片
- 生成向量并写入向量库
- 按用户问题进行相似检索
- 将检索结果作为上下文交给对应 Agent

你要完成的子任务：
- 文档导入器
- 文本切分器
- 向量化服务
- 检索服务
- 检索结果格式化器

建议接口：

```java
public interface RetrievalService {
    List<KnowledgeChunk> search(String query, int topK);
}
```

### 6. 接入向量数据库
计划文档给出的备选是 `Milvus` 或 `Redis Stack`。你至少要先打通一个。

要完成的功能：
- 建立向量集合或索引
- 存储文本切片及元数据
- 支持相似度检索
- 返回检索文本、来源、分数

如果时间紧：
- 第一阶段优先选容易接入的方案
- 先完成“能检索、能返回文本片段”的闭环

### 7. 设计异步任务机制
文档里明确提到 AIGC 类任务耗时，系统需要异步处理能力。即使你负责的是多智能体，也应该先把通用任务管理框架搭起来。

你至少需要：
- 提交任务接口
- 查询任务状态接口
- 任务执行器
- Redis 或内存状态缓存

建议状态：
- `PENDING`
- `RUNNING`
- `SUCCESS`
- `FAILED`

适合抽象成：

```java
TaskManager
TaskStatusService
AsyncJobExecutor
```

### 8. 提供前后端联调接口
前端要能接入你的多智能体服务，所以必须提供明确接口。

建议优先提供：
- `POST /api/chat`
  用于发起一次多 Agent 问答
- `GET /api/chat/{sessionId}`
  用于查询会话结果
- `POST /api/task`
  用于提交异步任务
- `GET /api/task/{taskId}`
  用于查询异步任务状态

如果要做流式输出，再增加：
- `WebSocket`
- SSE 接口

### 9. 实现最小可运行闭环
你的阶段性目标不是一次性做完整平台，而是先做通。

第一版闭环建议：
1. 用户提问
2. 后端收到请求
3. 调度三个 Agent
4. 每个 Agent 调用 RAG 检索
5. 每个 Agent 调用模型生成回答
6. 聚合器输出统一结果
7. 前端展示回答

只要这条链跑通，后面再优化。

## 你现在需要的前置条件

### 1. 技术选型确认
必须先确认这些内容，否则你会反复返工：
- 后端是否固定为 `SpringBoot`
- 是否使用 `Spring AI`
- 模型服务提供商是哪家
- 向量库用 `Milvus` 还是 `Redis Stack`
- 前端通过 HTTP、SSE 还是 `WebSocket` 联调

### 2. 知识库原始资料
没有资料就无法做 RAG。你至少需要：
- 非遗主题基础介绍
- 工艺流程资料
- 历史文化背景
- 可公开引用的文本内容

如果资料未整理，先要求团队输出：
- 原始文档
- 结构化摘要
- 可导入的 txt、md、pdf 内容

### 3. Agent 角色定义文档
你需要明确每个 Agent：
- 负责回答什么
- 不该回答什么
- 回答风格是什么
- 是否需要引用知识来源

### 4. 接口字段约定
要提前和前端、其他模块负责人对齐请求和响应结构，避免你写完后没人能接。

至少确认：
- 用户请求字段
- 会话 ID 规则
- Agent 返回结构
- 错误码
- 任务状态字段

### 5. 环境与配置
你需要准备：
- 模型 API Key
- Embedding 模型配置
- 向量库地址
- Redis 地址
- 本地开发配置文件

## 推荐开发顺序

### 第一阶段：单 Agent 跑通
目标：先别做复杂调度，先能完成一次“提问 -> 检索 -> 回答”。

要做：
- 建 SpringBoot 基础工程
- 接模型调用
- 接一个简单检索服务
- 返回一个答案

### 第二阶段：扩展为多 Agent
目标：把单 Agent 复制升级为多角色协作。

要做：
- 定义多个 Agent
- 增加调度器
- 增加回答聚合器
- 返回多角色结果

### 第三阶段：增加异步与状态管理
目标：让耗时任务可查询、可追踪。

要做：
- 引入 Redis 或状态缓存
- 实现任务提交与查询
- 支持失败重试或错误返回

### 第四阶段：前端联调
目标：让界面能真实调用你的服务。

要做：
- 补接口文档
- 联调聊天接口
- 联调状态查询
- 修复跨域、超时、结构不一致问题

## 验收标准
你负责的部分至少应满足以下标准：
- 用户输入一个问题后，系统能调用多个 Agent 进行回答
- 回答不是纯随机生成，而是带有知识库支持
- 系统接口可被前端调用
- 耗时任务有状态可查
- 发生异常时系统能返回明确错误，而不是直接崩溃

## 当前风险
- 没有知识库资料，RAG 无法落地
- 没有统一接口规范，前后端难以联调
- 先做复杂多轮协作会拖慢进度
- 没有任务状态管理会导致长耗时请求体验很差

## 建议
先做“能跑”的版本，再做“完整”的版本。  
对你来说，最重要的不是一步到位，而是先把以下闭环做出来：

`用户提问 -> 多 Agent 调度 -> RAG 检索 -> 模型生成 -> 聚合返回`

只要这条链跑通，你的主任务就真正开始落地了。
